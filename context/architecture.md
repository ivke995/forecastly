# Architecture

- Forecastly uses Next.js 16 with the App Router.
- The active route tree currently lives under the repository-root `app/` directory, not `src/app/`.
- Core app entry points are `app/layout.tsx`, `app/page.tsx`, and `app/globals.css`.
- The project uses TypeScript and Tailwind CSS.
- Repository-root `types/` contains erased TypeScript-only app-domain model contracts. `types/city.ts` defines normalized city models, and `types/weather.ts` defines normalized weather condition, current, hourly, daily, and aggregate forecast models that reference `City` through a type-only import.
- Repository-root `lib/` contains framework-light, UI-free external-data services and helpers that call Open-Meteo with `fetch` where needed, validate `unknown` provider payloads, normalize valid data into Forecastly domain contracts, and expose typed service errors for callers.
- `lib/geocoding.ts` queries the Open-Meteo Geocoding API, filters invalid result rows, normalizes valid rows into `CitySearchResult`, returns `[]` for empty/no-result searches, and throws `GeocodingError` for provider failures.
- `lib/coordinate-location.ts` converts browser geolocation coordinates into `CitySearchResult`-compatible objects for `getWeatherForecast`, validates latitude/longitude ranges, formats deterministic coordinate fallback display, attempts Open-Meteo reverse geocoding for readable display metadata, and returns coordinate fallback results when readable lookup fails.
- `lib/weather-codes.ts` centralizes WMO/Open-Meteo weather-code mapping in `getWeatherCondition(code: number)`, returning typed `WeatherCondition` values with stable emoji/severity metadata and a low-severity unknown fallback for unmapped finite numeric codes.
- `lib/weather-risk-badges.ts` derives stable current-weather risk badge metadata from normalized `CurrentWeather` fields without React, API calls, persistence, or provider request changes.
- `lib/open-meteo.ts` queries the Open-Meteo Forecast API for a seven-day forecast, validates current/hourly/daily provider sections, uses `getWeatherCondition` for current/hourly/daily weather-code normalization, normalizes valid data into `WeatherForecast`, skips malformed hourly/daily rows without returning partial domain objects, and throws `WeatherForecastError` for API/network/parse/malformed-payload failures.
- `app/layout.tsx` defines static Forecastly metadata and the shared root app shell.
- The shell composes generated shadcn sidebar primitives with `TooltipProvider`, `SidebarProvider`, `Sidebar`, `SidebarInset`, `SidebarTrigger`, and menu/group primitives. It renders text Forecastly branding, an active Dashboard/Search item, single-page section shortcuts for Current/Hourly/7-Day/Favorites, a disabled Settings item for a nonimplemented future destination, and a `main` content region constrained to `max-w-[1200px]` with responsive padding. The sidebar remains mobile-accessible through the official shadcn responsive sheet/trigger behavior.
- `app/page.tsx` is a `'use client'` component that integrates `CitySearch`, the `Use my location` browser geolocation flow, `useTemperatureUnit()` for persisted Celsius/Fahrenheit display, and `useFavorites()` for localStorage-backed favorite-city controls. City search and favorite selection call the same `handleCitySelect(city)` path, which sets selected city state and fetches a seven-day forecast with `getWeatherForecast(city)`. The location action validates browser support, calls `navigator.geolocation.getCurrentPosition()` with a 10-second timeout, resolves coordinates through `resolveCoordinateLocation()`, shows a non-blocking coordinate-fallback notice when needed, and fetches through `getWeatherForecast(location.city)`. On success, the page renders the city/location `displayName`, an accessible favorite star toggle reflecting `isFavorite(selectedCity.id)`, an accessible Celsius/Fahrenheit toggle, `CurrentWeatherCard`, `WeatherRecommendations`, `FavoriteCities`, `HourlyForecast`, and `DailyForecast`. Unit switching updates display props without refetching weather data.
- `app/globals.css` defines Tailwind v4 `@theme inline` mappings for shadcn-compatible CSS variables, Geist font variables, and app-wide weather-themed light/dark OKLCH tokens for backgrounds, foregrounds, cards, popovers, primary/secondary/accent surfaces, borders, inputs, rings, charts, and sidebar colors. It also imports `shadcn/tailwind.css` and `tw-animate-css` for shadcn/ui theming and animations.
- `components/ui/` houses reusable shadcn/ui component primitives (`Card`, `Button`, `Input`, `Separator`, `Sheet`, `Skeleton`, `Tooltip`, and the official `Sidebar` primitive set). These are installed via the shadcn CLI (`npx shadcn@latest add <component>`) and are the building blocks for feature-specific UI components.
- `@tsparticles/react` and `@tsparticles/slim` are installed as client-side animation dependencies for planned decorative weather backdrop components. The current package API exposes `Particles` plus `ParticlesProvider`/`useParticlesProvider`, with slim engine registration through `loadSlim(engine)` from `@tsparticles/slim`.
- `lib/utils.ts` provides the `cn()` utility (clsx + tailwind-merge) used by shadcn components and available for project-wide className merging.
- `components.json` at the repository root configures the shadcn/ui CLI (style, aliases, icon library, Tailwind settings).
