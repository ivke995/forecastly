# Architecture

- Forecastly uses Next.js 16 with the App Router.
- The active route tree currently lives under the repository-root `app/` directory, not `src/app/`.
- Core app entry points are `app/layout.tsx`, `app/page.tsx`, and `app/globals.css`.
- The project uses TypeScript and Tailwind CSS.
- Repository-root `types/` contains erased TypeScript-only app-domain model contracts. `types/city.ts` defines normalized city models, and `types/weather.ts` defines normalized weather condition, current, hourly, daily, and aggregate forecast models that reference `City` through a type-only import. `WeatherCondition` includes the numeric provider-compatible code, label, description, emoji, and severity. Runtime integrations map provider data into these contracts without mirroring raw API responses.
- Repository-root `lib/` contains framework-light, UI-free external-data services and helpers that call Open-Meteo with `fetch` where needed, validate `unknown` provider payloads, keep provider response interfaces local, normalize valid data into Forecastly domain contracts, and expose typed service errors for callers.
- `lib/geocoding.ts` queries the Open-Meteo Geocoding API, filters invalid result rows, normalizes valid rows into `CitySearchResult`, returns `[]` for empty/no-result searches, and throws `GeocodingError` for provider failures.
- `lib/coordinate-location.ts` converts browser geolocation coordinates into `CitySearchResult`-compatible objects for `getWeatherForecast`, validates latitude/longitude ranges, formats deterministic coordinate fallback display, attempts Open-Meteo reverse geocoding for readable display metadata, and returns coordinate fallback results when readable lookup fails.
- `lib/weather-codes.ts` centralizes WMO/Open-Meteo weather-code mapping in `getWeatherCondition(code: number)`, returning typed `WeatherCondition` values with stable emoji/severity metadata and a low-severity unknown fallback for unmapped finite numeric codes.
- `lib/weather-risk-badges.ts` derives stable current-weather risk badge metadata from normalized `CurrentWeather` fields without React, API calls, persistence, or provider request changes.
- `lib/open-meteo.ts` queries the Open-Meteo Forecast API for a seven-day forecast, validates current/hourly/daily provider sections, uses `getWeatherCondition` for current/hourly/daily weather-code normalization, normalizes valid data into `WeatherForecast`, skips malformed hourly/daily rows without returning partial domain objects, and throws `WeatherForecastError` for API/network/parse/malformed-payload failures.
- `app/layout.tsx` defines static Forecastly metadata and the shared root app shell.
- The shell renders a text-only Forecastly header above a `main` content region constrained to `max-w-[1200px]` with responsive padding.
- `app/page.tsx` is a `'use client'` component that integrates `CitySearch` for city selection and a shadcn `Button` for the `Use my location` flow. It manages `selectedCity`, `forecast`, `isLoading`, `isLocating`, `error`, and `locationNotice` state with `useState`/`useCallback`, and uses `useTemperatureUnit()` for persisted Celsius/Fahrenheit display state. On city selection, the page fetches a seven-day forecast via `getWeatherForecast`. On the location action, the page checks browser geolocation support, calls `navigator.geolocation.getCurrentPosition()` with a 10-second timeout, resolves coordinates through `resolveCoordinateLocation()`, shows a non-blocking notice when coordinate fallback display is used, and fetches the forecast through `getWeatherForecast(location.city)`. A loading spinner is shown during city or location fetches. Unsupported geolocation, permission denial, position unavailable, timeout, `CoordinateLocationError`, and `WeatherForecastError` paths surface styled readable error messages. On success, the city/location `displayName`, an outline shadcn `Button` for accessible Celsius/Fahrenheit toggling, the `CurrentWeatherCard` component, the `WeatherRecommendations` component, the `HourlyForecast` component (24 hourly entries in a horizontally scrollable card plus an hourly temperature/precipitation trend chart), and the `DailyForecast` component (7-day forecast in a responsive grid/scroll layout plus daily high/low temperature and precipitation trend chart) are displayed. The unit toggle calls the hook's `toggleUnit` function, updates display props immediately, preserves the selected forecast/city state, and does not refetch weather data. The current card maps `forecast.current` into flat props and falls back from missing apparent temperature to current temperature, missing relative humidity to `0`, and missing wind speed to `0`; it receives the selected `unit` and `getCurrentWeatherRiskBadges(forecast.current)` for optional current-risk badge rendering. The page also passes the selected `unit` to hourly and daily forecast components, and passes `getWeatherRecommendations(forecast)` into `WeatherRecommendations`, which renders only when the helper returns at least one item. Before any city/location is selected, the onboarding placeholder is visible.
- `app/globals.css` defines Tailwind v4 `@theme inline` mappings for shadcn-compatible CSS variables, Geist font variables, and app-wide weather-themed light/dark OKLCH tokens for backgrounds, foregrounds, cards, popovers, primary/secondary/accent surfaces, borders, inputs, rings, charts, and sidebar colors. It also imports `shadcn/tailwind.css` and `tw-animate-css` for shadcn/ui theming and animations.
- `components/ui/` houses reusable shadcn/ui component primitives (`Card`, `Button`, `Input`, `Separator`, `Sheet`, `Skeleton`, `Tooltip`, and the official `Sidebar` primitive set). These are installed via the shadcn CLI (`npx shadcn@latest add <component>`) and are the building blocks for feature-specific UI components.
- `lib/utils.ts` provides the `cn()` utility (clsx + tailwind-merge) used by shadcn components and available for project-wide className merging.
- `components.json` at the repository root configures the shadcn/ui CLI (style, aliases, icon library, Tailwind settings).
